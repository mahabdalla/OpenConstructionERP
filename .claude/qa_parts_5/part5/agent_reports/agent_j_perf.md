## Bottlenecks identified

- **[HIGH] SQLite lock contention on concurrent writes**: `35` 5xx out of `62` (`93.5%` error rate) when 20 clients POST to the same BOQ. Throughput capped at `0.1` pos/s.
- **[HIGH] Read latency degrades under write pressure**: projects p95 `15.2` ms → `22357.0` ms (`+146843.7%`) under 10 concurrent writers.
- **[MED] 200 concurrent HTTP/1.1 clients queue heavily**: no connection refusals/timeouts, but p95=`54251.3` ms vs baseline `~25` ms. Uvicorn is serializing requests behind the single-writer DB loop; client-side queueing masks as latency.
- **[HIGH] Concurrent POSTs to the same BOQ return 500 for ~60% of requests and hang the remainder past 30s**: confirms SQLite `database is locked` / busy-timeout exhaustion. On Postgres this workload should complete in <2s with no 5xx. Must be retested against Postgres before claiming production readiness.
- **[MED] Autocomplete p95 under 50 concurrent clients = `1570.8` ms (target 500 ms)** even with an EMPTY cost database. Root cause: uvicorn handling is serialized by the same DB access pattern used by writes. With CWICR fully loaded this will be worse.
- **[LOW] `/api/health` does not expose memory metrics**; external psutil probe required. Consider adding `rss_mb`, `gc_stats`, `db_pool_size` to health payload for operators.
