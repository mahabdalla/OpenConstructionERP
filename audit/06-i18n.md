## 4. Raw key-leak risk analysis

Pattern scanned: `t(\`prefix.${var}\` [, options])`.

- Total dynamic `t()` calls: **160**
- With `defaultValue` option: **160** (100%)
- Risky (no `defaultValue`, chained with `.replace`/`.toLowerCase` etc.): **0**

Examples of safe dynamic keys:
- `t(\`cde.state_${row.from_state}\`, { defaultValue: ... })` — CDEHistoryDrawer:99
- `t(\`catalog.category_${cat.toLowerCase().replace(/[^a-z0-9]+/g, '_')}\`, { defaultValue: cat })` — CatalogPage:1521
- `t(\`changeorders.reason_${order.reason_category}\`, { defaultValue: order.reason_category })` — ChangeOrdersPage:745

**Verdict**: raw key leaks are **not a problem** in the codebase. Every dynamic lookup has a fallback string, typically the raw value itself. This is defensively written. Top-10 risky entries requested — there are **zero** to list.

The only minor concern is bundle-size: 160 unique dynamic key patterns also means 160 buckets of suffix-keys in the fallbacks file that need parallel translation coverage across 20 locales — and as noted in §2 they often drift.

## 5. RTL Support

**Verdict**: READY.

Evidence:
- `frontend/src/app/i18n.ts:13` — Arabic declared with `dir: 'rtl'` marker.
- `frontend/src/app/App.tsx:260–280` — `useDocumentDirection()` hook sets `document.documentElement.dir` and `document.documentElement.lang` both on mount and on every `i18n.on('languageChanged', ...)` event.
- `frontend/src/shared/hooks/useIsRTL.ts` — exposes reactive `isRTL` to components.
- `frontend/src/index.css:173–295` — ~120 lines of `[dir='rtl']` overrides covering sidebar transform, border-l/r, padding/margin flips, chevron icon mirroring, text alignment, rounded corners, AG Grid container, and `body { direction: rtl }`.

Caveats:
- Only Arabic is marked RTL. Hebrew is not supported (not in `SUPPORTED_LANGUAGES`). Not a bug, just a scope note.
- `[dir='rtl']` handles a curated set of Tailwind utilities — any new CSS class using `ml-*`, `pl-*`, `left-*`, `rounded-l-*` etc. needs to be added to the overrides block manually. Tailwind's native `rtl:` variants are not being used; consider migrating when convenient.

## 6. Pluralization

**Verdict**: NOT IMPLEMENTED (risk: grammatically wrong strings in every count-dependent message).

Evidence:
- Zero i18next `_zero` / `_one` / `_two` / `_few` / `_many` / `_other` / `_plural` suffixes in `i18n-fallbacks.ts` for the actual i18next convention. (The 105 raw regex hits are **`*_other` keys used as ordinary keys** — e.g. `boq.cost_category_other: 'Other'` — not CLDR plural forms.)
- But `{{count}}` interpolation is used **853 times** in EN translations (e.g. `ai.estimate_complete_msg: '{{count}} items generated'`, `{{count}} files`, `{{count}} positions`).

Consequence: "1 items generated" will display for every count, including `1`. In Russian this breaks at 2, 5, 21, ... (three plural forms required). In Arabic six forms are required. Turkish is lenient, Chinese/Japanese/Korean have no plural, so they're unaffected. Realistically this mis-grammar affects EN/DE/FR/ES/PT/IT/PL/RU/AR/CS — 10 of the 20 locales.

## 7. Date / number / currency formatting

**Verdict**: WELL-DESIGNED, with two small leaks.

Positives:
- `frontend/src/shared/lib/formatters.ts:36` — `getIntlLocale()` central function maps i18next language → BCP-47 tag (e.g. `en → en-US`, `de → de-DE`, `ar → ar-SA`).
- `fmtNumber`, `fmtCompact`, `fmtCurrency`, `fmtDate` all use `Intl.*` with the mapped locale — no hardcoded `en-US`.
- ~90 call sites across `features/*` consistently use `getIntlLocale()` or `i18n.language`.
- `shared/ui/MoneyDisplay.tsx`, `shared/ui/DateDisplay.tsx`, `shared/ui/QuantityDisplay.tsx`, `features/settings/RegionalSettings.tsx` — all locale-aware.
- Gantt utilities (`shared/ui/Gantt/ganttUtils.ts`) and the Project Weather widget accept a `locale` parameter explicitly.

Leaks (use hardcoded locale instead of the mapped one):
- `features/boq/grid/columnDefs.ts:15` — `new Intl.NumberFormat('en', ...)` as fallback when `ctx?.fmt` missing (defensive, not user-facing but can leak if ctx is ever dropped).
- `features/boq/grid/cellRenderers.tsx:1546` — `ctx?.locale ?? 'en'` fallback.
- `features/boq/boqHelpers.ts:193` — `createFormatter(locale = 'de-DE')` defaults to German, not current locale. Harmless because all production call sites pass a locale, but a future caller could forget.
- `features/boq/VersionHistoryDrawer.tsx:146–151` and `features/dwg-takeoff/lib/pdf-export.ts:38` and `features/costs/EscalationCalculator.tsx:102` — use `Intl.NumberFormat(undefined, ...)` which means "browser default", *not* the user's UI language. For a Chinese user with a German browser this would format in German. Minor mismatch.
- `features/projects/ProjectDetailPage.tsx:184` — `formatDate(iso, locale = 'en-US')` — defaults to US English. Minor since the caller typically passes `i18n.language`.

## 8. Top 5 fixes needed

1. **Close the 20-locale translation drift for newly-added modules.** ~1,000 EN keys (BIM, CDE, Reports, Explorer, Integrations, Users, Onboarding, regional nav) are missing from 20 of 20 non-EN locales. Fix: regenerate bulk machine translations for missing-only keys per locale, then optionally flag uncertain ones for human review. Command could be: extract EN keys, diff vs each locale, ship through any LLM pipeline, merge back.

2. **Migrate translations out of `i18n-fallbacks.ts` (50k lines, shipped as a single JS chunk) into `frontend/public/locales/{code}/translation.json`.** The CLAUDE.md philosophy says "New language = JSON file" — currently it requires editing a giant TypeScript module. A switch to `i18next-http-backend` splits locales into parallel loadable chunks, shrinks initial bundle by ~90%, and brings the codebase in line with its own stated architecture.

3. **Replace all 15 hardcoded strings in `features/reports/ReportsPage.tsx`** (empty-state messages, section titles). This single file is responsible for ~25% of all hardcoded-string hits. Straightforward refactor — wrap each in `t('reports.empty.budget')` etc. and add the keys to EN fallbacks.

4. **Implement i18next pluralization for the 853 `{{count}}` usages.** At minimum add `_one`/`_other` variants for the handful of high-traffic messages (`ai.estimate_complete_msg`, `common.items_count`, BIM counts, catalog counts). i18next handles CLDR rules automatically once the suffixes are present. Without this, RU/AR/PL/CS users see grammatically wrong forms in every count-displaying toast and badge.

5. **Audit the ~5 `Intl.*(undefined, ...)` and hardcoded-locale formatter sites** (`boqHelpers.ts:193`, `VersionHistoryDrawer.tsx:146–151`, `dwg-takeoff/lib/pdf-export.ts:38`, `costs/EscalationCalculator.tsx:102`, `projects/ProjectDetailPage.tsx:184`). Route them through `getIntlLocale()` from `shared/lib/formatters.ts` to guarantee user's UI language drives number/date rendering.
